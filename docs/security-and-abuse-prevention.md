# Security and Abuse Prevention

## 1) Security goals

BRNK prioritizes:

- Safe link creation
- Fast but controlled redirect handling
- Abuse resistance (automation, spam, malicious URLs)
- Operational visibility and emergency controls
- User protection against open redirects and phishing

## 2) Security middleware layers

The `POST /shorten` flow applies layered defenses:

1. `express-rate-limit` fallback (30 req/15min per IP)
2. killswitch guard (blocks all creation during abuse spikes)
3. advanced creation rate limiter (per-IP, per-subnet, UA-based, progressive backoff)
4. URL safety inspection (IP URLs, nested shorteners, phishing patterns, domain trust scoring)
5. optional Google Safe Browsing validation (MALWARE, SOCIAL_ENGINEERING, UNWANTED_SOFTWARE, POTENTIALLY_HARMFUL_APPLICATION)
6. optional CAPTCHA challenge verification (reCAPTCHA v3, triggered only for suspicious requests)

This layered model prevents a single control failure from fully exposing the endpoint.

## 3) URL safety controls

URL safety middleware checks for:

- invalid/malformed URLs
- IP-based destination links (IPv4, IPv6, encoded IPs)
- nested shortener links (40+ known domains including bit.ly, tinyurl.com, t.co, snip.ly, shorte.st, etc.)
- suspicious phishing/malware patterns (login pages, credential harvesting, executable downloads)
- low trust domain characteristics

### Domain trust scoring (0–100)

Each destination URL receives a trust score based on:

| Factor | Effect |
|---|---|
| HTTPS | +10 |
| Suspicious TLD (.tk, .ml, .ga, .cf, .xyz, .click, etc.) | -30 |
| More than 3 subdomains | -10 |
| Any subdomain > 20 characters | -20 |
| Malware domain patterns (long random subdomains, consecutive digits) | -25 |
| URL > 500 characters | -10 |
| URL > 1000 characters | -10 (cumulative) |
| More than 10 query parameters | -10 |

URLs with trust scores below the configurable minimum (default 20) are rejected at creation time. URLs with trust scores below 50 or links created within the last 24 hours trigger an interstitial warning page for users.

If Google Safe Browsing API key is configured, additional external verdicting is applied. The system fails open — if the API is unreachable, links are allowed through to preserve availability.

## 4) Open redirect and QR code phishing mitigation

### Open redirect protection

Even with URL safety checks, malicious links can potentially bypass filters. BRNK addresses this with:

1. **Domain trust scoring**: Low-trust domains are rejected or flagged
2. **Interstitial warning page**: The frontend Redirect page fetches `/link-info/:shortCode` before redirecting. If the link has a low trust score or was recently created, users see a warning with the full destination URL and must explicitly confirm before proceeding
3. **Cache headers**: Permanent redirects (301/308) include `Cache-Control: public, max-age=86400, immutable` so browsers cache the redirect destination, reducing repeated exposure to the redirect service
4. **Backend-first redirects**: In production, Vercel routes `/:shortCode` directly to the backend, which returns HTTP redirect status codes without frontend involvement. The frontend interstitial is a fallback safety layer

### QR code phishing protection

QR codes generated by BRNK point to the same short URLs that go through the full security pipeline. Low-trust destinations trigger the same interstitial warning when the QR code is scanned and the link is followed.

## 5) Rate limiting and anti-automation

Rate limits are enforced at multiple dimensions:

### Per-IP limits
- Creation: configurable (default 5/min, 50/hour)
- General endpoints: 100 req/15min

### Per-subnet limits
- /24 subnet (IPv4) or /48 (IPv6): default 30/min, 200/hour
- Prevents distributed attacks from nearby IPs

### User-Agent throttling
Suspicious user agents (bots, curl, wget, python-requests, scrapy, headless browsers, selenium, puppeteer) receive 50% of normal rate limits.

Detection uses a single pre-compiled regex for performance:
```
/^$|curl|wget|python-requests|httpie|scrapy|bot|spider|crawl|headless|phantom|selenium|puppeteer/i
```

### Progressive backoff
Repeated rate limit violations trigger escalating blocks:

| Violations | Block duration |
|---|---|
| 3 | 5 minutes |
| 6 | 30 minutes |
| 10 | 1 hour |
| 20 | 24 hours |

### Global rate limiting (kill switch)
When the malicious link detection threshold is crossed (default: 10 flagged links in 5 minutes), the kill switch activates and blocks ALL new link creation requests until cooldown expires (default: 15 minutes).

This reduces scripted abuse while preserving legitimate usage. Botnets generating massive links would trigger both per-subnet limits and the global kill switch.

## 6) Killswitch behavior

Monitoring tracks malicious link events over a sliding window.
If threshold is crossed, killswitch activates and blocks further shorten requests temporarily.

Key env controls:

- `KILLSWITCH_MALICIOUS_THRESHOLD` (default: 10)
- `KILLSWITCH_WINDOW_MS` (default: 300000 = 5 minutes)
- `KILLSWITCH_COOLDOWN_MS` (default: 900000 = 15 minutes)

## 7) Monitoring endpoint protection

`/monitoring/dashboard` can be protected by setting `MONITORING_API_KEY`.
When set, requests must include `x-api-key` header.

**Key management recommendations**:
- Use a strong, randomly generated API key (32+ characters)
- Rotate the key periodically
- Do not expose the key in client-side code or public repositories
- Monitor for unauthorized access attempts in server logs
- Consider additional IP allowlisting at the infrastructure level for sensitive deployments

## 8) Proxy and anonymiser detection

The security middleware detects:

- **Excessive proxy hops**: Requests with > 5 IPs in `X-Forwarded-For` are blocked
- **Anonymising headers**: `via`, `x-proxy-id`, `forwarded`, `x-tor-is`, `x-tor`
- **Data center IPs**: Common private/cloud IP ranges (10.x, 172.16-31.x, 192.168.x)

This metadata is attached to `req.securityMeta` and used by downstream middleware (CAPTCHA triggering, rate limit adjustment).

## 9) Frontend and browser-side protections

- Redirect/error pages use constrained flows and validation of short code patterns
- Helmet adds security headers globally in backend
- CORS origin can be locked to known frontend domain
- Interstitial warning page for low-trust links prevents blind redirects
- QR codes use the same URL safety pipeline as direct links

## 10) API versioning

All API responses include `X-API-Version: 1.0` header. Clients should check this header to detect breaking changes in future API versions.

## 11) Production hardening recommendations

1. Always set strict `FRONTEND_URL` (avoid wildcard in production)
2. Always set `MONITORING_API_KEY` with a strong random value
3. Enable reCAPTCHA keys and Safe Browsing key for higher-risk deployments
4. Monitor dashboard trends and alerts daily (especially `redirectsLastMinute`, `recentFlaggedLinks`, `killSwitch` status)
5. Keep dependencies patched and run regular `npm audit` reviews
6. Keep redirect/status behavior documented for support and incident response
7. Configure rate limit environment variables appropriate to expected traffic
8. Set up external alerting on monitoring dashboard metrics for abuse trends

## 12) Privacy and data notes

BRNK stores short link metadata and click counts required for operation.
Do not store sensitive credentials in URLs because links may be logged by clients or intermediaries.
